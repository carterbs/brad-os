# Research Question

How `docs/quality-grades.md` is generated in `brad-os`, with focus on `/scripts/ralph` and related scripts: command path, scoring formula, and evidence used.

# Summary

`docs/quality-grades.md` is generated by `scripts/update-quality-grades.ts`, exposed as the npm command `npm run update:quality-grades`. The script can also be run directly with `npx tsx scripts/update-quality-grades.ts`.

`/scripts/ralph` does not directly execute the grade generator. It references `docs/quality-grades.md` as planning input for backlog refill prompts, where Ralph prioritizes tasks that raise grades.

# Detailed Findings

## Command path

- `package.json` defines `update:quality-grades` as `tsx scripts/update-quality-grades.ts`.
- The generator script writes to `docs/quality-grades.md`.
- The script has a direct usage comment: `npx tsx scripts/update-quality-grades.ts`.

## Ralph relationship

- Ralph imports and invokes `buildBacklogRefillPrompt` during backlog refill.
- That prompt explicitly instructs the agent to read `docs/quality-grades.md` and prioritize grade-improving tasks.
- No direct invocation of `update-quality-grades.ts` from `scripts/ralph/index.ts` was found.

## Scoring formula (implemented)

Baseline grade is coverage-first (and iOS test level influences top bands):
- coverage >=95: A (or B+ if iOS low)
- >=90: B+ (or B)
- >=80: B (or B-)
- >=70: C+
- >=60: C
- >=50: C-
- else D

Penalties/adjustments:
- Shared domain baseline fixed at B-
- High-risk untested files: degrade 2 sub-grades
- Any untested file: degrade 1 sub-grade
- API partial (`apiComplete !== Yes`): degrade 1 sub-grade
- Assertion density adjustment:
  - density >=2.0 and >=10 test cases: improve 1 sub-grade
  - density <1.0 with tests present: degrade 1 sub-grade

## Evidence inputs used by the script

- Domain mapping dictionaries:
  - handlers -> domain
  - services -> domain
  - repositories -> domain
  - iOS test class names -> domain
- Backend test evidence:
  - scans `packages/functions/src` for `.test.ts`
  - classifies handler/service/repo/integration/schema tests
  - counts test cases (`it`/`test`) and assertions (`expect`)
- iOS evidence:
  - scans both `ios/BradOS/BradOSCore/Tests/BradOSCoreTests` and `ios/BradOS/BradOSTests`
  - maps filenames by explicit table, then fallback name heuristics
- Untested file evidence:
  - detects handlers/services without corresponding unit or integration tests
  - assigns risk using high/medium/low pattern lists
- Coverage evidence:
  - optionally refreshes with `npm run test:coverage -- --reporter=dot --silent`
  - reads `packages/functions/coverage/coverage-summary.json`
  - aggregates line coverage by domain from handlers/services/repositories
- Curated markdown evidence:
  - extracts existing `## Active Tech Debt` and `## Recently Completed`
  - optionally updates those sections plus short domain notes via `claude -p --model sonnet` using computed metrics

# Architecture Insights

Grade generation is deterministic up through the computed metrics (tests, coverage, untested files, risk, density). Narrative notes and tech-debt/recently-completed section updates may be AI-assisted, but guarded by fallback to existing curated content if Claude output fails.
